・サーブレットを文字コードに対応させるためには、「フィルタ」を使用する。

　何らかの処理を行う機能のこと。

・プッシュした際に「URIが検出できませんでした」みたいな事が表示されてプッシュできないことについて、
　原因はやはりtxtファイルにアクセストークンを記載していたことだった。そのため、該当コミットを修正すると
　無事にプッシュできた。「ヒストリー」から該当コミットの1つ前のコミットを選択し、右クリックで「リセット」→「混合」を選択。
　混合だと変更がステージングされない状態でそのコミットまで戻ることができる。ソフトはステージングされた状態。
　ハードは現在からそのコミット時点までの変更が全て削除されてしまうので、使う際は注意する。
　参考サイト　https://mmmnn1257.hatenablog.com/entry/2018/10/28/213313

・クラスファイルがデフォルトの場所になく、ビルドしても作成されない
　　→ 恐らく、javaファイルを置く場所を間違えていたか、そもそもeclipseで表示されなかっただけかもしれない。
　　　 Windowsのエクスプローラーを確認すると、ちゃんと「RequestRecvP\build\classes\sample_pkg02」配下に
　　　 クラスファイルが作られていた。javaファイルを置く場所に関してはプロジェクトのビルドパスの
　　　 「ソース」の「ビルドパス上のソースフォルダー」にjavaファイルが無ければならない、と推測する。

・HTMLファイルからサーブレットに移行しない
　　→ 原因は、HTMLファイルをエクスプローラーからダブルクリックで開いていることだった。
　　　 eclipseから「実行」→「サーバーで実行」で実行し、フォームを送信したら無事にサーブレットに移行した。

　　　 クラスファイルの位置が正しくないことが原因だと思い、クラスファイルの場所を調べたり配置を変えてみたりしたが、
　　　 webapp\WEB-INF配下のclassesフォルダにクラスファイルが無く、デフォルトで作成される場所である
　　　 「RequestRecvP\build\classes\sample_pkg02」配下にだけある状態でもちゃんとHTML画面から
　　　 サーブレットに移行したので、クラスファイルはこのままでいいと思われる。

・サーブレットファイルの内容を変更したのに、ブラウザ上でサーブレットの内容が変更されない
　　→ 原因はtomcat9(java 11)ではなくtomcat9(java17)のサーバーを使ったこと？
　　　 元々ブラウザでの実行はjava11の方を指定していたが、java17の方でも変わらずに動くのか試したくなり
　　　 変えてみたところこのようになってしまった。なぜtomcatのバージョンを変えると変更が適用されないのか
　　　 まだわからないのでこれから調べていきたい。
